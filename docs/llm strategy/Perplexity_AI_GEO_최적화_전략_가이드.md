## 1. Perplexity AI 개요 및 특성

### 1.1 Perplexity의 정체성

Perplexity AI는 스스로를 **'답변 엔진(Answer Engine)'**으로 정의합니다. 전통적인 검색 엔진이 링크 목록을 제공하는 것과 달리, Perplexity는 **웹에서 수집한 정보를 종합하여 단일 답변을 생성**하고, 각 정보의 출처를 명확하게 인용합니다.

### 1.2 다른 LLM과의 차별점

| 특성 | Perplexity | ChatGPT | Gemini | Claude |
|------|-----------|---------|--------|--------|
| **핵심 성격** | 학술/연구 도구 | 대화형 AI | 멀티모달 통합 | 안전 중심 AI |
| **인용 수 (응답당)** | 6.61개 | 2.62개 | 6.1개 | 다양 |
| **콘텐츠 톤 선호** | 객관적/학술적 | 대화형 | 종합적 | 명확/간결 |
| **갱신 민감도** | 매우 높음 (2-3일) | 분기별 | 정기적 | 정기적 |

### 1.3 Perplexity가 인용하는 주요 도메인

Perplexity는 **수동으로 큐레이션된 권위 있는 도메인 목록**을 유지합니다:

- **Reddit**: 전체 인용의 6.6% (최다 인용 도메인)
- **Wikipedia**: 학술/사실 정보
- **GitHub**: 기술/개발 콘텐츠
- **Stack Overflow**: 프로그래밍 관련
- **LinkedIn**: 전문가/비즈니스
- **Coursera**: 교육 콘텐츠
- **학술지**: 연구/논문

## 2. Sonar 검색 시스템 아키텍처

### 2.1 Sonar API 개요

Perplexity의 핵심 검색 시스템인 **Sonar**는 실시간 웹 검색과 자연어 처리를 결합한 **RAG(Retrieval Augmented Generation) 아키텍처**를 사용합니다.

**작동 방식:**
1. 수천억 개의 웹페이지를 실시간으로 인덱싱
2. 문서를 **세분화된 하위 단위(청크)**로 분리
3. 각 청크에 개별 점수를 부여
4. 최종 답변 생성 시 고점수 청크를 조합

### 2.2 Sonar 모델 종류

| 모델 | 특성 | 컨텍스트 크기 | 용도 |
|------|------|-------------|------|
| **Sonar (기본)** | 빠른 처리 | ~127K 토큰 | 빠른 Q&A |
| **Sonar Pro** | 심층 분석 | 확장됨 | 복잡한 다단계 쿼리, **2배 더 많은 인용** |
| **Sonar Reasoning-Pro** | 자율 추론 | 확장됨 | 다단계 추론이 필요한 복잡 질문 |

### 2.3 검색 프로세스 흐름

```
사용자 질문 입력
       ↓
[1단계] 초기 검색 - 기존 검색과 유사한 방식으로 결과 수집
       ↓
[2단계] 청크 분할 - 문서를 하위 단위로 분리
       ↓
[3단계] L3 리랭킹 - ML 필터로 품질 재평가
       ↓
[4단계] 품질 임계값 검사 - 기준 미달 시 전체 폐기
       ↓
[5단계] 답변 생성 - 고점수 청크 조합 + 인용 첨부
       ↓
최종 답변 출력
```

## 3. 인용 선택 메커니즘 (L3 리랭킹 시스템)

### 3.1 3단계 리랭킹 시스템

Perplexity는 **L3(3-Layer) 리랭킹 시스템**을 통해 인용 소스를 선별합니다:

| 단계 | 명칭 | 설명 |
|------|------|------|
| **1단계** | 초기 검색 | 전통적 검색 엔진과 유사한 방식으로 결과 검색 및 초기 점수화 |
| **2단계** | L3 ML 리랭커 | 머신러닝 필터 적용, 엔티티 검색에 더 엄격한 기준 |
| **3단계** | 품질 임계값 | 최소 기준 미달 시 해당 결과 세트 전체 폐기 |

### 3.2 리랭킹 시 고려되는 요소

**긍정적 요소:**
- 원본 연구/데이터 포함
- 전문가 인용 및 권위 있는 출처 언급
- 명확한 Q&A 구조
- 짧은 단락, 글머리 기호
- 최신 게시/수정일

**부정적 요소:**
- 과도한 마케팅/홍보 문구
- 모호하고 감성적인 표현
- 오래된 콘텐츠 (2-3일 이후 감쇠 시작)
- 키워드 스터핑

## 4. 핵심 인용 랭킹 요인 분석

### 4.1 요인별 가중치

| 요인 | 가중치 | 상세 설명 |
|------|--------|----------|
| **출처 신뢰도** | 높음 | 원본 연구, 전문가 인용, 정부/학술 기관 출처 |
| **도메인 권위** | ~15% | 신뢰성, 전문성, 신뢰도의 복합 점수 |
| **콘텐츠 구조** | 높음 | 명확한 제목, Q&A 형식, 글머리 기호, 짧은 단락 |
| **인용 위치** | ~20% | 시각적 인용 배치가 랭킹 가중치에 영향 |
| **스키마 마크업** | ~10% | 구조화된 데이터로 AI 이해도 향상 |
| **콘텐츠 신선도** | 높음 | 게시 후 **2-3일** 이후 콘텐츠 감쇠 시작 |

### 4.2 콘텐츠 신선도의 중요성

Perplexity는 **콘텐츠 신선도에 매우 민감**합니다:

```
게시 후 0-2일: 최고 가중치
게시 후 3-7일: 감쇠 시작
게시 후 7일+: 지속적 감쇠
오래된 콘텐츠: 시간 민감 쿼리에서 제외 가능
```

**대응 전략:**
- "최종 업데이트: YYYY년 MM월 DD일" 형식으로 날짜 명시
- 2-3일 주기로 콘텐츠 갱신 일정 수립
- 시간에 민감하지 않은 상시 콘텐츠(evergreen)도 정기 갱신

### 4.3 주제별 가중치 부스팅

Perplexity는 특정 주제에 대해 **최대 3배의 가중치**를 부여합니다:

- **높은 가중치**: 기술, AI, 비즈니스 콘텐츠
- **중간 가중치**: 교육, 건강 정보
- **표준 가중치**: 일반 정보성 콘텐츠

## 5. 콘텐츠 구조 최적화 전략

### 5.1 답변 우선(Answer-First) 원칙

Perplexity가 선호하는 콘텐츠 구조의 핵심은 **즉각적인 답변 제공**입니다:

```
✅ 권장 구조:
┌─────────────────────────────────────────┐
│ [H1 제목]                               │
├─────────────────────────────────────────┤
│ [직접적인 답변] ← 80토큰(약 40-60단어) 이내 │
├─────────────────────────────────────────┤
│ [근거/이유] ← "왜냐하면" 라인에 숫자/단위   │
├─────────────────────────────────────────┤
│ [상세 설명] ← 구조화된 섹션들             │
└─────────────────────────────────────────┘
```

### 5.2 최적 제목 구조

**질문 형식의 H2, H3 제목 사용:**

```html
<!-- 권장 예시 -->
<h2 id="comparison">갱신형 vs 비갱신형 암보험, 무엇이 좋을까?</h2>
<h2 id="premium">암보험 월 보험료는 얼마가 적정한가?</h2>
<h3 id="age-30">30대 암보험 추천 상품은?</h3>

<!-- 피해야 할 예시 -->
<h2>암보험 안내</h2>
<h2>상품 정보</h2>
```

**딥링킹용 `id` 속성:**
- 모든 주요 제목에 `id` 속성 추가
- Perplexity가 특정 섹션을 직접 인용할 수 있도록 지원

### 5.3 콘텐츠 길이 권장사항

| 콘텐츠 유형 | 권장 길이 | 비고 |
|------------|----------|------|
| 종합 가이드 | 10,000단어 이상 | 토픽 클러스터 형태 |
| 비교 분석글 | 3,000-5,000단어 | 표/차트 포함 |
| FAQ 페이지 | 2,000-3,000단어 | 질문당 100-200단어 |
| 뉴스/업데이트 | 500-1,500단어 | 신선도가 더 중요 |

**가독성 점수:**
- Flesch 가독성 점수 **55 이상** 유지
- 짧은 문장, 명확한 표현 사용

### 5.4 의도 블록(Intent Block) 설계

Perplexity가 인식하기 쉬운 **의도 블록** 유형:

| 의도 유형 | 콘텐츠 형식 | 예시 |
|----------|-----------|------|
| **정의** | "X는 Y이다" 형태 | "비갱신형 암보험은 보험 기간 동안 보험료가 고정되는 상품이다." |
| **단계** | 1-2-3 순차 | "암보험 가입 절차: 1) 상담 신청 → 2) 건강 고지 → 3) 청약서 작성" |
| **비교** | X vs Y | "갱신형 vs 비갱신형: 갱신형은 초기 보험료가 낮고, 비갱신형은 장기 비용이 유리" |

## 6. E-E-A-T 신호 구현 방법

### 6.1 E-E-A-T 개요

| 요소 | 영문 | 의미 | 구현 방법 |
|------|------|------|----------|
| **E** | Experience | 경험 | 실제 경험 기반 콘텐츠 작성 |
| **E** | Expertise | 전문성 | 자격증, 경력 명시 |
| **A** | Authoritativeness | 권위 | 공신력 있는 출처 연결 |
| **T** | Trustworthiness | 신뢰성 | HTTPS, 투명한 정보, 수정 이력 |

### 6.2 E-E-A-T 구현 예시

**Experience (경험):**
```
"10년차 보험설계사로서 1,000건 이상의 보험금 청구를 처리한 경험을 바탕으로..."
"직접 보험금을 청구하고 3주 만에 수령한 실제 경험을 공유합니다..."
```

**Expertise (전문성):**
```html
<div class="author-bio">
  <strong>작성자:</strong> 김○○
  <ul>
    <li>손해사정인 자격증 보유</li>
    <li>AFPK(재무설계사) 자격</li>
    <li>보험업계 15년 경력</li>
  </ul>
</div>
```

**Authoritativeness (권위):**
```
- 금융감독원 공시 자료 인용
- 보험연구원 통계 연결
- 금융위원회 발표 내용 참조
- 국가암등록통계 데이터 활용
```

**Trustworthiness (신뢰성):**
```html
<meta name="last-modified" content="2024-12-04">
<div class="update-info">
  최종 업데이트: 2024년 12월 4일
  수정 이력: 2024년 암 발생률 통계 반영
</div>
```

### 6.3 Perplexity가 신뢰하는 권위 도메인

Perplexity는 **카테고리별 권위 도메인 목록**을 관리합니다:

| 카테고리 | 권위 도메인 예시 |
|---------|----------------|
| 기술/개발 | GitHub, Stack Overflow, MDN |
| 전문/비즈니스 | LinkedIn, Coursera, HBR |
| 학술/사실 | Wikipedia, 학술지, .edu 도메인 |
| 한국 보험 | 금융감독원, 보험연구원, 금융위원회 |

**전략:** 이들 도메인에서 브랜드/웹사이트가 언급되도록 **디지털 PR** 수행

## 7. 기술적 SEO 설정

### 7.1 PerplexityBot 이해

Perplexity는 **두 가지 크롤러**를 운영합니다:

| 크롤러 | 목적 | AI 훈련 사용 |
|--------|------|-------------|
| **PerplexityBot** | 검색 결과용 웹 콘텐츠 인덱싱 | ❌ 아니오 |
| **Perplexity-User** | 실시간 사용자 쿼리 지원 | ❌ 아니오 |

### 7.2 robots.txt 설정

```txt
# Perplexity 크롤러 허용 설정
User-agent: PerplexityBot
Allow: /

User-agent: Perplexity-User
Allow: /

# 선택적: 크롤 속도 제한
User-agent: PerplexityBot
Crawl-delay: 1
```

### 7.3 IP 주소 화이트리스트

Perplexity는 크롤러 IP 정보를 JSON 형태로 공개합니다:

- **PerplexityBot**: `https://www.perplexity.com/perplexitybot.json`
- **Perplexity-User**: `https://www.perplexity.com/perplexity-user.json`

방화벽/보안 설정 시 해당 IP 범위를 화이트리스트에 추가하세요.

### 7.4 기술적 필수 요구사항

| 요구사항 | 설명 | 중요도 |
|---------|------|--------|
| **Google 인덱싱** | Perplexity 가시성의 전제조건 | 필수 |
| **클린 XML 사이트맵** | 모든 중요 페이지 포함 | 필수 |
| **강력한 내부 링킹** | 관련 페이지 간 연결 | 높음 |
| **Core Web Vitals 준수** | LCP, FID, CLS 기준 충족 | 높음 |
| **모바일 최적화** | 반응형 디자인 | 필수 |
| **SSR/Static 렌더링** | 클라이언트 사이드 렌더링 피하기 | 권장 |

### 7.5 llms.txt 파일 구현 (선택)

LLM 친화적 콘텐츠 색인을 위한 새로운 표준:

```txt
# llms.txt - LLM 친화적 사이트 설명

# 사이트 설명
name: 보험가이드
description: 한국 보험 상품 비교 및 분석 전문 사이트

# 주요 콘텐츠
topics:
  - 암보험 추천 및 비교
  - 갱신형 vs 비갱신형 분석
  - 보험금 청구 가이드

# 전문성
expertise:
  - 손해사정인 자격 보유 전문가 집필
  - 금융감독원 공시 데이터 기반
```

## 8. 스키마 마크업 구현

### 8.1 FAQPage 스키마

Perplexity가 가장 효과적으로 인식하는 스키마:

```json
{
  "@context": "https://schema.org",
  "@type": "FAQPage",
  "mainEntity": [
    {
      "@type": "Question",
      "name": "암보험 진단금 적정 금액은 얼마인가요?",
```

6. 구조화된 데이터(스키마) 구현

GEO는 일회성 프로젝트가 아닌 지속적인 프로세스입니다. AI 검색 알고리즘은 빠르게 진화하므로, 정기적인 모니터링과 전략 업데이트가 필수입니다.
